## What is MCP in This Generator

MCP (Model Context Protocol) is a standardized protocol for exposing server-side tools to LLM clients. This generator emits a complete TypeScript MCP server project from a plain-English tool description.

**In this module's context:**
- The generator takes a tool description (e.g., "Tool that looks up customer records by email")
- It outputs a runnable MCP server project with `McpServer` + `StdioServerTransport`
- The server exposes the tool via stdin/stdout using the MCP protocol
- LLM clients (Claude Desktop, etc.) can invoke the tool through the MCP handshake

### `src/server.ts`

The core MCP server implementation. Key elements:

```typescript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { z } from 'zod/v4';

const server = new McpServer({
  name: 'tool-that-looks-up-customer-records-by-email',
  version: '1.0.0',
}, {
  capabilities: { logging: {} },
  instructions: 'Customer lookup tool. Always validate email format before searching.',
});

server.registerTool(
  'tool_that_looks_up_customer_records_by_e',
  {
    description: 'Look up customer records by email address. Validates email format.',
    inputSchema: z.object({
      email: z.string().email().describe('Customer email address'),
    }),
  },
  async ({ email }, { timeoutMs }) => {
    // ... implementation
  }
);

async function main(): Promise<void> {
  const transport = new StdioServerTransport();
  await server.connect(transport);
}

main();
```

**Role:** Initializes `McpServer`, registers the tool, and connects the `StdioServerTransport`. This is where the MCP protocol handshake happens over stdio.

## MCP SDK Usage

### `McpServer`

The central class from `@modelcontextprotocol/sdk`:

```typescript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';

const server = new McpServer(
  { name: 'my-server', version: '1.0.0' },
  {
    capabilities: { logging: {} },
    instructions: 'Always validate email format before searching.',
  }
);
```

**Constructor params:**
- `name` + `version` — server identity
- `capabilities` — what the server supports (`logging`, `resources`, `prompts`, etc.)
- `instructions` — guidance for the LLM client (shown in system prompt)

### `registerTool()`

Registers a tool with Zod validation:

```typescript
server.registerTool(
  'tool_name',
  {
    title: 'Human-readable title',
    description: 'What the tool does',
    inputSchema: z.object({ ... }),
    outputSchema: z.object({ ... }),
    annotations: { readOnlyHint: true, idempotentHint: true },
  },
  async ({ arg1, arg2 }, { timeoutMs }) => {
    return {
      content: [{ type: 'text', text: 'result' }],
      structuredContent: { key: 'value' },
    };
  }
);
```

**Key points:**
- `inputSchema` — Zod schema for args; MCP SDK validates before calling the handler
- `structuredContent` — machine-readable response alongside the text
- `timeoutMs` — optional per-call timeout override (falls back to `TIMEOUT_MS` env var)
- `isError` — set to `true` in `content` to signal error responses

### `StdioServerTransport`

The transport for local, process-spawned servers:

```typescript
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';

const transport = new StdioServerTransport();
await server.connect(transport);
```

**How it works:**
- Parent process spawns the server via `node server.js`
- Communication happens over `stdin`/`stdout` using the MCP wire protocol
- No network ports needed — works with Claude Desktop and other stdio-based clients

## Tool Schema and Validation

### Zod Input Validation

The generated server validates all tool inputs with Zod:

```typescript
inputSchema: z.object({
  email: z.string().email().describe('Customer email address'),
}),
```

**Email shape enforcement:**
- `z.string().email()` — validates format `user@domain.tld`
- Manual regex check as secondary guard: `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
- Returns `isError: true` with `"Invalid email format"` for bad input

### Structured Response Behavior

The handler returns both `content` (for humans) and `structuredContent` (for machines):

```typescript
return {
  content: [{ type: 'text', text: JSON.stringify(customer, null, 2) }],
};
```

Clients can parse `structuredContent` if they prefer machine-readable output. The text form degrades gracefully.

## Safe Defaults

All generated servers follow these rules:

| Rule | How It's Enforced |
|------|-------------------|
| **No hardcoded secrets** | All sensitive config via env vars (`PORT`, `TIMEOUT_MS`, `LOG_LEVEL`) |
| **Input validation** | Zod schema + manual email regex; rejects malformed input before processing |
| **Try/catch** | `try/catch/finally` in handler and `lookupCustomerByEmail`; errors return `isError: true` |
| **Timeouts** | `setTimeout` in `lookupCustomerByEmail`; configurable via `TIMEOUT_MS` env var |
| **No raw PII logging** | Logger uses `emailLength` (not email) and masked IDs (`Buffer.from(email).toString('base64').slice(0,12)`) |
| **Safe error messages** | Internal errors hidden from clients; only user-safe messages returned |
| **No secrets in response** | `customer` object only returns `email` (user-provided), `id`, `found`, `message` — no tokens or internal data |

## Local Usage Flow

### Step 1: Generate the Demo

```bash
cd backend/generator
npm install --legacy-peer-deps  # Windows workaround for @modelcontextprotocol/sdk
npm run generate:demo
```

Output: `generated/customer-lookup-mcp/`

### Step 2: Inspect Generated Output

```bash
ls -la generated/customer-lookup-mcp/
cat generated/customer-lookup-mcp/src/server.ts
```

Review the generated `server.ts` and `package.json` before running.

### Step 3: Install and Run the Generated Server

```bash
cd generated/customer-lookup-mcp
npm install --legacy-peer-deps
npm run build        # Compile TypeScript
node dist/server.js  # Start server (runs on stdio)
```

The server is now available via MCP stdio. It will listen on stdin for tool calls.

### Step 4: Use MCP Inspector (Optional)

The MCP SDK provides an interactive inspector for testing servers locally:

```bash
cd generated/customer-lookup-mcp
npx @modelcontextprotocol/inspector
```

This opens an interactive prompt where you can:
- List available tools
- Invoke tools with sample inputs
- Inspect server capabilities and responses

## Extension Notes for Day 2

Day 2 will integrate Bob AI to make generated servers **dynamic** — the template remains the fallback and safety baseline.

### What Changes

| Aspect | Day 1 (Template) | Day 2 (Bob) |
|--------|-----------------|-------------|
| `generationMode` | `'template'` | `'bob'` |
| Tool description | Hardcoded | From Bob AI |
| `generateMcpServer.ts` | Ignores description | Wires description to Bob |
| `complexity.ts` | Static rules | Bob-informed risk |
| Tests | 50 passing | Add Bob paths |

### Safety Rules Remain

**All Day 1 safety defaults must survive Day 2:**
- No hardcoded secrets
- Zod input validation
- Try/catch everywhere
- `TIMEOUT_MS` with abort timer
- No raw PII in logs
- `structuredContent` in responses

If Bob returns an unsafe template, the generator should **reject and fall back** to the Day 1 template rather than emit dangerous code.
